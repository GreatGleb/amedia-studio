# Технические задания и спецификации фич (/specs)

В этой папке хранятся технические задания (ТЗ) и подробные спецификации для функционала нашего проекта. Наличие четкого ТЗ позволяет ИИ-ассистентам безошибочно и быстро реализовывать задачи, соответствующие высоким стандартам нашего кода.

## Структура папки `/specs/`

- **/templates/** — Шаблоны для создания новых спецификаций.
- **`[feature-name].md`** — Файлы ТЗ для конкретных фич или модулей (например, `portfolio-localization.md`, `contact-form-validation.md`).

> [!NOTE]
> Придерживайтесь принципа **DDD (Domain-Driven Design)** на фронтенде: если ТЗ касается крупного бизнес-домена (например, панель поставщика, профиль пользователя), спецификации можно группировать по подпапкам.

---

## Правила написания ТЗ для ИИ

Чтобы ИИ-ассистент (Antigravity, Roo Code) максимально точно реализовал задачу, ТЗ должно содержать следующие ключевые разделы:

### 1. Бизнес-цели и контекст (Business Value)
*Зачем мы это делаем? Какую проблему решает фича?*
Пример: *"Оптимизировать конверсию формы обратной связи за счет добавления интерактивной валидации полей в реальном времени."*

### 2. Функциональные требования (Functional Requirements)
*Что конкретно должно работать и как вести себя при взаимодействии?*
- Описание пользовательских сценариев (User Flows).
- Описание состояний (загрузка, ошибка, пустая выдача, успех).

### 3. Нефункциональные требования и UI/UX (Design & Performance)
*Как это должно выглядеть и работать на уровне ощущений?*
- Адаптивность (мобильные экраны от 320px, планшеты, десктопы).
- Анимации (плавность, использование GSAP / Framer Motion, обязательное очищение эффектов при размонтировании).
- Производительность (отсутствие скачков разметки, оптимизация изображений).

### 4. Технический стек и ограничения (Tech Specs)
- Использование React 19 (новые Actions, `use()`, отсутствие устаревших хуков).
- Tailwind CSS 4 (новые CSS-переменные и встроенные утилиты).
- TypeScript (строгая типизация всех пропсов и данных).

### 5. Критерии приемки (Acceptance Criteria)
*Четкий чек-лист, по которому ИИ и разработчик проверяют готовность задачи.*
Пример:
- `[ ]` Форма отправляет данные по API маршруту `/api/contact`.
- `[ ]` Поля валидируются по событию `onBlur`.
- `[ ]` Анимация отправки длится не более 300мс и не лагает.

## Как ИИ должен работать со спецификациями

1. **Изучение ТЗ:** Перед началом любой задачи ИИ обязан найти и прочитать соответствующий файл ТЗ в `/specs/`.
2. **Создание плана:** На основе ТЗ создается план реализации (`implementation_plan.md`) в соответствии с `ai/workflows/ai-dev-flow.md`.
3. **Отслеживание прогресса:** В ходе разработки ИИ отмечает выполненные пункты требований.
